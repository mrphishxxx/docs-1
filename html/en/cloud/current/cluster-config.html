Cluster Configuration

What Settings are Available and What Do They Mean?

Introduction

Once you have signed in on Elastic Cloud Console, three main categories of settings are available:

  • Cluster settings
  • Plugin / Bundle settings (Elastic Cloud Premium customers only)
  • Account settings

In this section we will address the cluster configuration page where you may change the cluster settings. Once signed in to the Elastic Cloud Console, access it by selecting the configuration link below an existing cluster in the left sidebar or click create new cluster. The only difference when creating a new cluster or configuring an existing one is that the former allows to specify the region for the cluster to reside within.

Region

When choosing a region the general rule is to choose one as close to your application servers as possible in order to minimize the network delay.

you can only select region when you create a new cluster. Region cannot be changed later.

Capacity and SSD

Depending upon how much data you got and what queries you will execute you will have to select a cluster capacity. Unfortunately, there is no silver bullet for deciding how much memory you will need other than just simply testing it. Fortunately, you can change the capacity of the cluster at a later time, without any downtime. More details on how much memory is needed in the blog article, Elasticsearch in Production.

Fault Tolerance

The chosen capacity, as described above, is per data center. The reason for this is that there is no point in having two data centers if one failing will result in a cascading error because the remaining center cannot handle the total load. You may choose between one, two, or three data centers that all no shared single point of failure beyond residing within the same city. Through the allocation awareness in Elasticsearch, we configure the nodes so that your cluster will automatically allocate replicas between each data center.

After selecting region, capacity, and number of data centers, you get the price per hour and accumulated per month. The monthly price is only for convenience, as billing is done by the hour. This implies that you can test a bigger or smaller capacity and only pay for the hours used.

Version

At any given time, the two latest minor versions will be available for deployments of new clusters (e.g., Elasticsearch 1.6 and 1.5 or 2.0 and 1.6). The latest patch level within a minor version is available for new deployments (e.g., 2.0.1 and 1.6.5).

Selecting a Version

If you are creating a new cluster the default version is preselected, and unless there are any known issues this one corresponds to the default download from elastic.co.

Upgrade

When changing the version of an existing cluster, either a major or a minor upgrade will be performed. The essential difference is that a minor upgrade (e.g. from 1.5 to 1.6) will have no downtime while a major upgrade (e.g. from 1.6 to 2.0) will require the cluster to stop.

Customers are given 6 months to upgrade existing clusters when a minor Elasticsearch version is removed for deployment.

Dynamic Scripts

The script_fields in filters and facets are one of the features that make Elasticsearch so flexible, but they do allow arbitrary code execution, like Runtime.exec("cat /etc/passwd") and other malicious things. If you plan on delegating access to other users or allow any form of anonymous access then we recommend to disable dynamic scripts. More on this in Securing your Elasticsearch cluster.

Custom Plugins / Dictionaries / Scripts

If you have uploaded any plugins or user bundles with dictionaries or scripts then this where you choose to enable them for the cluster.

Only Premium customers have access to uploading custom plugins/bundles.

Plugins

This section has the list of official plugins available for the selected Elasticsearch version. When selecting a plugin from this list you get a version that has been tested with the chosen Elasticsearch version. The main difference between selecting a plugin from this list versus uploading the same plugin as a custom plugin is in who decides the version used. The reason we do not list the version chosen on this page is because we reserve the option to change it when necessary. That said, we will not force a cluster restart for a simple plugin upgrade unless there are severe issues with the current version. In most cases, plugin upgrades are applied lazily, in other words when something else forces a restart like you changing the plan or Elasticsearch runs out of memory.

Kibana 4

In order to use Kibana, your cluster must be running Elasticsearch version 1.6 or newer. By default, Kibana 4 is disabled. To start using Kibana 4, simply click on the Enable button, and within seconds you will see the URL which gives you access to Kibana 4.

Next, you need to provide the Kibana 4 endpoint with credentials so that you can access Elasticsearch resources, for example, granting read access to relevant Elasticsearch indexes and granting update access to the .kibana index.

Name

This setting allows you to assign a more human-friendly name to your cluster which will be used for future reference in the Elastic Cloud Console. Common choices are dev, prod, test, or something more domain specific.

Applying Changes

If you’re creating a new cluster all that now remains is to click the create new cluster button and the cluster will be provisioned. If you are working with an existing cluster then there are update buttons under each section. You can, however, do changes in any section and all changes will be applied at once, when you click update, like changing the capacity and upgrading to a new Elasticsearch version in one step.
